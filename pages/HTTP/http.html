# 五层模型介绍

# 网络协议分层

  • 应用层--http,ftp
  • 传输层--TCP/IP协议,UDP
  • 网络层: 为数据在节点之间传输创立逻辑链路
  • 数据链路层: 在通信的实体间建立数据链路服务
  • 物理层: 定义物理设备如何传输数据

# 传输层TCP/IP

  • 向用户提供可靠的端到端服务
  • 传输层向高层屏蔽了下层数据通信的细节

# 应用层

  • 为应用软件提供了很多服务
  • 构建与TCP之上
  • 屏蔽了网络传输相关细节

# http协议的发展历史

# http/0.9

  • 该版本极其简单，只有一个命令GET
  • 没有HEADER等描述数据的信息
  • 服务器发送完毕，就关闭TCP连接
  • GET /index.html命令表示，TCP 连接（connection）建立后，客户端向服务器请求（request）网页index.html。
  • 协议规定，服务器只能回应HTML格式的字符串，不能回应别的格式。
<html>
  <body>Hello World</body>
</html>

# http/1.0

  • 除了GET命令，还引入了POST命令和HEAD命令
  • 新增功能:处理请求的状态码status code、header发送接收数据的描述和操作
  • 多字符集支持、多部分发送（multi-part type）、权限（authorization）、缓存（cache）、内容编码（content encoding）等
  • 任何格式的内容都可以发送,可以传输文字，还能传输图像、视频、二进制文件
  • HTTP请求和回应的格式也变了,除了数据部分,每次通信都必须包括头信息（HTTP header,用来描述一些元数据
  • 请求格式:
GET / HTTP/1.0
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10_10_5)
Accept: */*
第一行是请求命令，必须在尾部添加协议版本（HTTP/1.0）。后面就是多行头信息，描述客户端的情况

  • 这些数据类型总称为MIME type，每个值包括一级类型和二级类型，之间用斜杠分隔。

2.6 缺点:

  • HTTP/1.0 版的主要缺点是，每个TCP连接只能发送一个请求。
  • 发送数据完毕，连接就关闭，如果还要请求其他资源，就必须再新建一个连接。
  • TCP连接的新建成本很高，需要客户端和服务器三次握手，并且开始时发送速率较慢（slow start）。
  • 所以，HTTP 1.0版本的性能比较差。随着网页加载的外部资源越来越多，这个问题就愈发突出了。
  • 为了解决这个问题，有些浏览器在请求时，用了一个非标准的Connection字段。
  • Connection: keep-alive字段要求服务器不要关闭TCP连接，以便其他请求复用。
  • 服务器同样回应这个字段。 Connection: keep-alive一个可以复用的TCP连接就建立了，直到客户端或服务器主动关闭连接。
  • 但是，这不是标准字段，不同实现的行为可能不一致，因此不是根本的解决办法。

# http/1

  • 持久链接
  • pipeline同个连接发送多个请求
  • 服务端按序发送接收
  • 增加host:可以在同一个物理服务器可以部署不同的web服务,提高效率

# http/2

  • 所有数据二进制传输
  • 同一个链接里面发送多个请求不再需要按顺序,并行返回多个数据,提高传输效率
  • 头信息压缩以及推送等提高效率的功能(推送服务端可以主动发送数据,请求html时可以把html中需要的css和js文件主动发送到客户端)

# http三次握手

  • 首先创建TCP connection 链接客户端和服务器的连接通道一直保持

  • http请求和响应都是数据包

  • 一个链接可发送多个请求

  • http三次握手创建了TCP链接,用于传输数据

  • 三次握手是为了防止开启无用的链接

  • 规避延时导致的服务器开销问题

  1. 客户端--> SYN Seq = 0 -->服务端
  2. 服务端--> SYN Seq = 0 Ack = (Seq+1) -->客户端
  3. 客户端-->Ack Seq = (Ack) Ack = (Seq+1) -->服务端

# URI URL URN

# URI

  • 统一资源标识符
  • 用来唯一标识互联网上的信息资源
  • 包括URL和URN

# URL

  • 统一资源定位器
  • http://user:pass@host.com:80/path?query=string#hash
  • http:// 通过什么协议访问服务
  • user:pass@ 特定身份访问,不常用,不安全
  • host.com 服务器在互联网的位置
  • :80 端口,定位host找到的服务器存放的某一个web服务器
  • /path 路由,/代表根目录
  • query=string搜索参数
  • hash 请求的片段,可做锚点

# URN

  • 永久统一资源定位符
  • 在资源移动后还可以找到

# HTTP code

  • 定义服务器对请求的处理结果
  • 100-199 操作持续进行,还有操作需要完成
  • 200-299 操作是成功的
  • 300-399 需要传递对象,其它方式获取
  • 400-499 发送的请求有问题
  • 500-599 服务器的问题
